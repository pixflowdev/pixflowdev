# Neumorphism Layouts

Use these as layout patterns. Adapt copy, counts, and section order to the run brief.

## Tactile App Dashboard

Purpose: make the recipe recognizable immediately.

Structure:

- Left rail or compact topbar with raised logo and inset active navigation.
- Main content region with a large raised overview panel.
- Metric cards in a two-by-two or three-card row.
- Chart panel with inset plotting area.
- Schedule, task, or activity panel with raised rows.
- Primary teal action placed as a raised control.

Rules:

- The first viewport should look like a usable interface, not a decorative landing hero.
- On desktop, the shell must fill the available canvas after the rail. Use a dominant overview column plus a supporting column/grid for metrics, controls, schedule, or approvals.
- Do not place a narrow dashboard stack in the center of a wide empty page.
- Use inset wells for inputs, chart tracks, selected nav, and progress areas.
- Keep enough spacing between raised cards for shadows to breathe.
- Do not use hard borders as the main separation method.

## Centered Hero With Dashboard Mockup

Purpose: support product landing pages while keeping the tactile system visible.

Structure:

- Compact centered header.
- Hero copy in a raised panel or above a raised device/dashboard mockup.
- CTA row with one raised teal primary and one neutral raised secondary.
- Large dashboard mockup showing inset panels, metrics, toggles, and chart tracks.

Rules:

- The dashboard mockup should be inspectable, with labels and states relevant to the brief.
- The mockup should be wide enough to read as a product surface. If centered, flank it with useful proof, metrics, or settings rather than leaving empty desktop whitespace.
- Avoid generic abstract rectangles.
- The hero can be simpler than the app shell, but the neumorphic material must be obvious in the first viewport.

## Settings Or Control Surface

Purpose: show interaction state and practical controls.

Structure:

- Raised settings panel.
- Inset text inputs, sliders, toggles, or segmented controls.
- Selected options use inset treatment plus teal accent.
- Status rows and confirmation panel use raised cards.

Rules:

- Make state transitions visible: default, selected, pressed, disabled, and focus.
- Use native controls where possible and keep labels programmatic.
- Do not make every element inset; reserve inset for controls that are physically pressed or carved into the surface.

## Final CTA Panel

Purpose: close with the same material language.

Structure:

- Wide raised panel with heading, short lead, and action row.
- Optional inset email/project field.
- Secondary support tile or status pill.

Rules:

- Do not switch to a normal flat CTA section.
- CTA controls should keep the raised/inset state model.
