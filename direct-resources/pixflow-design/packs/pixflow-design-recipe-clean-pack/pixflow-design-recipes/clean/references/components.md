# Clean Components

## Header

Use at the top of the page.

Fields:

- Brand wordmark or simple logo
- 3 to 4 nav links
- Sign-in text link
- Primary rounded CTA

Styling:

- White background, optional bottom border only if needed.
- Max-width container, generous horizontal padding.
- Hide center links on small screens before wrapping makes the header messy.
- Keep button labels short.
- Do not add a large hamburger unless the user asks for full mobile navigation.

## Buttons And CTAs

Use in the hero and final CTA.

Fields:

- Platform/action label
- Optional small icon
- Primary or secondary action state

Rules:

- Keep labels short enough for mobile.
- Use one primary filled CTA and one quieter secondary CTA.
- App-store style buttons are allowed only when the page is genuinely app-oriented.
- Button height should be at least 44px.
- Secondary buttons use a border or white fill, not a second strong accent.
- Focus-visible state must be obvious.

## Product Slab / Device

Fields:

- Product name or UI area
- Status, amount, progress, preview, or key object
- 3 to 5 visible UI rows or controls
- Optional accent stripe, chip, or avatar

Rules:

- Product visuals must be readable without zooming.
- Tilt/offset only the outer card; never rotate text enough to hurt readability.
- Keep border radius consistent with the page.
- Use a stable aspect ratio.
- Product proof should include enough real labels that the user understands the workflow.
- Prefer one strong product slab over several decorative fake cards.

## Code / Command Mockup

Use when the brief is developer, AI, plugin, automation, or technical workflow.

Fields:

- Window chrome or tab strip
- Sidebar or file list
- Code/command lines
- Suggestion, result, diff, or status panel
- Bottom status row when useful

Rules:

- Keep line lengths readable and avoid overflowing the mockup.
- Use monospace only inside code/command zones.
- Use accent sparingly for selected tabs, active rows, and primary suggestions.

## Feature Card

Fields:

- Simple icon
- Title
- One short sentence
- Optional tiny supporting label or mini UI state

Rules:

- Three-card rows are acceptable, but two to four stronger cards are usually better.
- Avoid large decorative icons.
- Keep each card purpose distinct.
- Add a small UI state only when it helps explain the feature.
- Cards should use borders first and shadows second.
- Text should not exceed 65 characters per line in a large card.

## Pricing Card

Fields:

- Plan name
- Audience sentence
- Price or package summary
- Included points
- CTA
- Optional recommended badge

Rules:

- Keep pricing honest. If the brief does not include real prices, use placeholder ranges or non-financial plan labels.
- Highlight only one card.
- Badges must stay inside the card edge.
- Buttons must align visually across cards where possible.

## Testimonial Card

Fields:

- Short quote or proof result
- Name/role or anonymous proof label
- Optional avatar initials
- Divider

Rules:

- Keep quotes believable.
- Do not fabricate endorsements from real companies.
- Avoid oversized quotation marks.

## Integration Tile

Fields:

- Short product/platform/category label
- Optional two-letter mark or icon tile
- Optional secondary row of pill labels

Rules:

- Tiles should fit their text without clipping.
- Use muted color for non-action marks.
- On mobile, two columns are usually enough.

## Trust Or Security Band

Fields:

- Circular icon/emblem
- Reassurance heading
- One short explanation
- Optional three proof chips

Rules:

- Use high contrast in the dark band.
- Do not make security or compliance claims without source material.
- If the design does not need a dark band, use a pale proof or CTA band instead.
